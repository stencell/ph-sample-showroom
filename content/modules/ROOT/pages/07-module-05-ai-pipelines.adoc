Running evaluations manually in a notebook is a starting point, not a production practice.
If nobody remembers to run the evaluation, the quality gate does not exist.
If the notebook fails silently, nobody knows.

This module shows how to package the evaluation workflow from Module 4 into a reproducible, schedulable AI Pipeline on Red Hat OpenShift AI using Kubeflow Pipelines v2.
You will review a pre-built pipeline definition, compile it, submit a run, schedule it as a nightly recurring job, and confirm that evaluation results flow back into MLflow.
At the end, the full AgentOps loop is in place and running automatically.

*Estimated time*: 15 minutes

* Configure a nightly recurring run schedule for automated evaluation
* Confirm that pipeline-triggered evaluation results are linked to MLflow runs

== Exercise 1: Anatomy of an AI evaluation pipeline

Before submitting a run, you will read the pipeline definition to understand how the evaluation logic from Module 4 was packaged as containerized components.

@dsl.component(base_image="quay.io/agentops-lab/eval-runner:latest")
def compare_and_gate(v1_metrics: dict, v2_metrics: dict, threshold: float) -> str:
    """Compares two evaluation results. Returns 'pass' or 'fail'."""
    delta = v2_metrics["similarity"] - v1_metrics["similarity"]
    return "pass" if delta > -threshold else "fail"

@dsl.pipeline(name="agentops-prompt-evaluation")
def evaluation_pipeline(prompt_version: int = 1, threshold: float = 0.10):
    baseline_task = run_evaluation(
        prompt_version=1,
        experiment_name="agentops-lab"
    )
    eval_task = run_evaluation(
        prompt_version=prompt_version,
        experiment_name="agentops-lab"
    )
    gate_task = compare_and_gate(
        v1_metrics=baseline_task.output,
        v2_metrics=eval_task.output,
        threshold=threshold
    )
----

. Note three things about this pipeline:

* The evaluation logic inside `run_evaluation` is identical to Module 4 -- the `@dsl.component` decorator wraps it without rewriting it
* Pipeline parameters (`prompt_version`, `threshold`) make the pipeline reusable for any prompt version comparison, not just v1 vs v2
* The `compare_and_gate` component encodes the quality gate as code -- reviewable, version-controlled, and auditable in the same way as application code

=== 1.2 Review the base container image

The `@dsl.component` decorator specifies `base_image="quay.io/agentops-lab/eval-runner:latest"`.

This image packages the same Python dependencies as your workbench -- `mlflow`, `langchain`, `kfp`, and the `agentops_client` library.

In a production setup, you would maintain this image alongside your application code in a CI/CD pipeline: when a new dependency is added to the workbench, the image is rebuilt and the pipeline automatically picks it up on the next run.

NOTE: The `eval-runner` image is pre-pulled to the cluster nodes during environment provisioning to avoid cold-pull latency during the lab.

=== 2.5 Schedule a nightly recurring run

. Click *Create recurring run*. Configure:

[cols="1,2",options="header"]

| Pipeline
| `agentops-prompt-evaluation`

| Trigger type
| Periodic

| Interval
| Every 1 day

| Start time
| Tomorrow at 02:00 UTC

| `prompt_version`
| `1`

| `threshold`
| `0.10`
|===
+
Click *Create*.
+
The nightly evaluation is now scheduled.
Each night at 02:00 UTC, the pipeline will evaluate the current production prompt against itself (v1 as both baseline and candidate) -- a sanity check that the baseline itself has not drifted.
In production, you would parameterize the run to test any new candidate prompt that is proposed for deployment.

NOTE: The recurring run schedule is stored in the RHOAI database and persists across cluster restarts.
It is scoped to your lab instance -- you are not scheduling against a shared system.
