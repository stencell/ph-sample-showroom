[cols="1,3",options="header"]
|===
| Agent | Role

| *Supervisor*

== Exercise 2: The three pillars and the ownership map

Understanding *which tool answers which question* is what separates reactive incident response from proactive observability.
This exercise maps the three pillars to the tools in your environment and establishes who is responsible for each layer.

=== 2.1 The three pillars

[cols="1,2,2",options="header"]
|===
| Pillar | What it tells you | Tool in this environment

| *Metrics*
| Quantitative signals over time -- token rate, latency percentiles, error rate
| Grafana + Prometheus

| *Logs*
| Structured event records per container -- what happened and when
| LokiStack

| *Traces*
| End-to-end request journeys across agent spans -- where time was spent, what failed silently
| MLflow
|===

Each pillar reveals a different class of failure:

* Metrics tell you *something is wrong* -- error rate is spiking, latency is above your SLO
* Logs tell you *what happened* -- which container logged the error and what the message was
* Traces tell you *where time was spent* -- which agent in the pipeline was slow or produced empty output

No single pillar gives you the full picture.
All three are required for production reliability in a multi-agent system.

=== 2.2 The ownership map

Two roles share the observability stack but own different layers.

[cols="1,2,2",options="header"]
|===
| | *Platform and SRE team* | *AI developer team*

| *Metrics*
| Infrastructure health, SLO dashboards, alerting
| Token usage, cost per request (shared visibility)

| *Logs*
| Pod restarts, OOM kills, node events
| Application events, tool call results, retrieval failures

| *Traces*
| Not applicable -- infrastructure has no trace instrumentation
| Span timelines, latency attribution, silent failures

| *Accountability*
| Uptime, capacity, networking
| Trace quality, prompt versioning, evaluation pipelines
|===

Both teams collaborate during incident response.
Neither team can diagnose a multi-agent failure alone.

== Exercise 3: Send your first query

Now you will submit a live query to the AI Ops Assistant from the pre-loaded RHOAI workbench notebook.
Note the round-trip latency -- this becomes your baseline for the failure discovery in module 2.

=== 3.1 Open the workbench notebook

. Navigate to the OpenShift AI dashboard at link:{rhods_dashboard_url}[{rhods_dashboard_url}^].

. In the left navigation, select *Data Science Projects*.

. Click on the *AgentOps Lab* project.

. Under *Workbenches*, find the *agentops-lab* workbench and click *Open*.
+
The workbench opens in a new tab.

. Locate the file `agentops-lab.ipynb` and click to open it.

=== 3.2 Run the environment setup cell

. Find and run *Cell 1: Environment setup*.
+
This cell imports required libraries and confirms the environment variables are configured:
+
[source,python]
----
import os
import time
from agentops_client import AgentOpsAssistant

# Verify environment
print("MLflow tracking URI:", os.getenv("MLFLOW_TRACKING_URI"))
print("MaaS API base:", os.getenv("MAAS_API_BASE"))

assistant = AgentOpsAssistant()
print("Assistant initialized successfully")
----
+
Expected output:
+
----
MLflow tracking URI: http://mlflow-agentops-demo.apps.cluster.example.com
MaaS API base: https://maas.example.com/v1
Assistant initialized successfully
----

=== Verify

Both environment variables print a URL -- not `None`.

If you see `None` for either variable, the workbench environment was not configured correctly.
Notify the facilitator before continuing.

=== 3.3 Send a query and note the latency

. Run *Cell 2: Send a query*:
+
[source,python]
----
start = time.time()
response = assistant.query("How do I configure node affinity in OpenShift AI?")
latency = time.time() - start

print(response)
print(f"\nRound-trip latency: {latency:.1f}s")
----

. The cell prints the assistant's response and the round-trip latency.
+
Expected output (latency will vary):
+
----
You can configure node affinity in OpenShift AI by adding a nodeSelector or
affinity rule to your Notebook or ServingRuntime resource...

Round-trip latency: 4.2s
----

. *Note this latency value.* Write it down or remember it -- you will look for this exact request in traces and logs in later modules.

=== Verify

* The response text is a coherent answer, not an empty string or error message
* A round-trip latency in seconds is printed
* Latency is between 2 and 15 seconds (higher values are worth noting)

If the response is empty or you see an exception, run the cell a second time.
Occasional empty responses are expected -- one of the pre-seeded failure conditions causes intermittent empty results from the runbook retrieval agent.
You will diagnose this in module 3.
