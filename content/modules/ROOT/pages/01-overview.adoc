== The situation

Your platform team just got paged.
The AI Ops Assistant -- a multi-agent system your organization uses to answer platform engineering questions -- is responding slowly.
One developer reports empty answers.
Another says latency is through the roof.
Nobody knows which agent is failing.

Sound familiar?

Multi-agent AI systems fail distributedly.
A timeout in a retrieval agent silently produces empty results.
A misbehaving prompt floods downstream agents with noise.
Token costs spike with no visible cause.
Standard monitoring catches none of it -- you need metrics, logs, and traces working together across every agent in the pipeline.

That is what *AgentOps* is: the operational discipline that brings observability end-to-end across an agentic AI system.
This workshop teaches you that discipline hands-on, using a live application with pre-seeded failure conditions you will discover and diagnose.

== The lab application: AI Ops Assistant

The AI Ops Assistant is a four-agent LangGraph system deployed in your cluster, right now, handling live traffic.

[cols="1,3",options="header"]
|===
| Agent | Role

| *Supervisor*
| Receives user queries and routes them to the appropriate retrieval agent using an LLM call via MaaS

| *Docs retrieval*
| Queries a ChromaDB vector store pre-loaded with RHOAI and OpenShift documentation; uses MaaS for response generation

| *Runbook retrieval*
| Queries a ChromaDB vector store pre-loaded with SRE runbooks and troubleshooting guides; uses MaaS for response generation

| *Synthesis*
| Combines results from retrieval agents and calls MaaS for a final coherent response
|===

A load generator has been submitting realistic platform engineering queries since the environment was provisioned.
By the time you arrive, Grafana dashboards are populated, logs are flowing, and MLflow has recorded hundreds of traces.
You will not wait for data -- the data is already there.

The application has two intentionally seeded failure conditions.
You will discover them in modules 2 and 3.

== Learning objectives

By the end of this workshop, you will be able to:

. Identify the components of a multi-agent AI architecture and explain why distributed observability matters for production reliability
. Distinguish the three pillars of observability -- metrics, logs, and traces -- and map each to specific tools in the Red Hat AI stack
. Navigate Grafana dashboards to analyze AI-specific metrics: token usage, inference latency, and tool call success rates
. Write and execute LogQL queries in LokiStack to filter and correlate logs across containers in a multi-agent application
. Instrument a LangGraph application with MLflow autolog for zero-code-change distributed tracing
. Analyze MLflow trace timelines to identify latency bottlenecks and silent tool failures in multi-agent workflows
. Use the MLflow Prompt Registry to version-control system prompts and link them to evaluation results
. Design evaluation datasets using deterministic and LLM-as-a-Judge scorers to assess agent output quality
. Detect prompt regressions by comparing evaluation results across prompt versions before production deployment
. Package notebook-based evaluations into reproducible, schedulable AI Pipelines on OpenShift AI

== Success criteria

You have successfully completed this workshop when you can:

* Read a Grafana AI dashboard and answer "which agent is the slowest?" and "what is my cost per request?" from live data
* Write a LogQL query that isolates errors from a specific agent container
* Add MLflow autolog to a LangGraph notebook and verify a trace appears in the MLflow UI
* Identify a latency bottleneck and a silent failure in a real trace timeline
* Run a prompt evaluation, compare two versions, and interpret the regression result
* Submit an AI Pipeline run for a scheduled evaluation job
